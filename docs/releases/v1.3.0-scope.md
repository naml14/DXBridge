# DXBridge v1.3.0 Scope

## Release intent

`v1.3.0` is planned as a compatibility-preserving minor release for the DXBridge `1.x` line.

The release goal is compatible growth:

- deliver one clearly additive piece of public value
- improve internal hardening around the already-published `1.x` surface
- finish the docs and release framing needed to publish the change without ambiguity

This document closes the intended scope for `v1.3.0`. Work outside this list does not belong in the release unless this document is updated first.

## Baseline

- Repository version baseline at planning time: `1.2.0`
- Public ABI baseline to preserve: `include/dxbridge/dxbridge.h` + `include/dxbridge/dxbridge_version.h`
- Export baseline to preserve: `src/dxbridge.def` ordinals `1` through `43`
- Compatibility target: existing `1.0.x`, `1.1.x`, and `1.2.x` consumers keep working without source or binary changes unless they explicitly opt into new `v1.3.0` additions

## Closed objectives

### Public new value

`v1.3.0` may add exactly one public growth area:

1. Add an additive capability-discovery surface so bindings and native consumers can query backend or device compatibility details without guessing from backend selection side effects.
2. Keep that new surface optional and documentation-first: existing consumers must be able to ignore it and continue using the current `1.x` flow unchanged.
3. Document how the new capability information should be used before device, swap-chain, and pipeline setup.

### Internal hardening

`v1.3.0` should strengthen reliability around the existing API surface, not redefine it.

1. Tighten validation around current lifecycle-sensitive paths: initialization, adapter enumeration, device creation, swap-chain resize/present, and frame begin/end boundaries.
2. Expand regression coverage for stale handles, null pointers, `struct_version` checks, and wrong-order calls on existing APIs.
3. Add or improve ABI-focused checks so existing exports, ordinals, type widths, and struct layouts are verified before release.

### Docs and release work

`v1.3.0` must also land the release-facing material needed to explain the change cleanly.

1. Publish a release note split that clearly separates public additive value, internal hardening, and docs or release-process work.
2. Add a compatibility note stating that `v1.3.0` is a minor release for `1.x`, not a contract reset.
3. Update consumer-facing docs so the new capability-discovery story and the existing compatibility guarantees are easy to find.

## Explicit compatibility promise for v1.3.0

`v1.3.0` is not allowed to:

- remove or rename any existing exported `DXBridge_*` function
- change existing export ordinals in `src/dxbridge.def`
- resize, reorder, or repurpose fields in existing public structs
- change numeric values of existing result codes, handles, enums, or flags
- change the width of opaque handles from `uint64_t`
- change the process-wide backend model of `DXBridge_Init()`
- change the thread-local error retrieval model of `DXBridge_GetLastError()` and `DXBridge_GetLastHRESULT()`
- make old call sequences require new mandatory parameters, callbacks, or setup steps

## Out of scope

The following items are explicitly excluded from `v1.3.0`:

- new rendering backends beyond DX11 and DX12
- a redesign of the exported C ABI or a move away from the current handle model
- changes that require `DXBRIDGE_STRUCT_VERSION` to be bumped for existing structs
- removal or semantic repurposing of `DXBridge_SupportsFeature()`
- async, multithreaded, or multi-instance backend lifetime redesign
- descriptor-management, resource-binding, or command-recording overhauls
- shader-model expansion, advanced pipeline feature work, or major rendering-surface growth unrelated to capability discovery
- generated bindings or broad example rewrites across every language runtime
- platform expansion beyond the current Windows desktop target

## Acceptance gates for PR2

PR2 should be considered in scope only if it satisfies all of these:

1. Any public API addition is additive and optional.
2. Existing `1.x` consumers remain source-compatible and ABI-compatible.
3. Existing exports keep their current names and ordinals.
4. Tests or checks cover the non-breaking guarantees, not only the new feature.
5. Release notes and docs present the change in three buckets: public value, hardening, and docs or release work.

## Implementation notes for the next PR

- Prefer one coherent capability-query family over several small one-off helpers.
- Preserve the meaning of current APIs even if a richer replacement is added.
- Treat docs and ABI verification as release-blocking work, not cleanup that can slip.
