# DXBridge v1.3.0 Compatible Design

## Purpose

This document defines the compatibility guardrails and extension pattern for `v1.3.0`.

It does not authorize a broad API expansion. It describes how `v1.3.0` may grow while staying inside the published `DXBridge 1.x` contract.

## Compatibility-first design rules

### 1. Existing binary exports stay stable

The current exported surface defined by `src/dxbridge.def` is the locked baseline for `1.x`.

- Keep every existing export name unchanged.
- Keep ordinals `1` through `43` unchanged.
- If `v1.3.0` adds exports, append them after the current list.
- Do not replace an old export with a new meaning under the same name.

### 2. Existing public types stay byte-for-byte compatible

The current public header contract in `include/dxbridge/dxbridge.h` remains authoritative.

- Existing handle typedefs remain `uint64_t`.
- `DXBResult` remains `int32_t`.
- Enum-like public categories remain `uint32_t`-based.
- Existing constants keep their numeric values.
- Existing structs keep the same field order, field meaning, and alignment expectations.
- `struct_version` stays at offset `0` for all current public structs.

### 3. Existing runtime semantics stay stable

`v1.3.0` must preserve the observable behavior that current bindings already rely on.

- `DXBridge_Init()` remains process-wide backend selection.
- `DXBridge_SupportsFeature()` keeps its current active-backend semantics; it must not be silently redefined into a global machine capability query.
- `DXBridge_GetLastError()` and `DXBridge_GetLastHRESULT()` remain thread-local error accessors.
- Destroy functions remain safe compatibility helpers for existing callers; they must not become failure-reporting APIs that require new handling.
- `DXBridge_GetBackBuffer()` continues to return a swap-chain-owned render target handle rather than a newly owned resource.

### 4. New public surface must be additive and opt-in

If `v1.3.0` introduces a public API, the addition must follow these rules:

- old source code compiles unchanged against the new headers
- old binaries continue to work against the new DLL
- new APIs are optional convenience or capability helpers, not new mandatory setup steps
- new behavior defaults do not alter existing call paths unless the caller explicitly chooses the new API

## Recommended extension pattern for v1.3.0

### Single capability-discovery family

The preferred design for the `v1.3.0` public addition is one coherent capability-discovery family, instead of many narrow helpers.

The family should answer questions that today require guesswork, especially for bindings and portable sample code, such as:

- what backend or adapter capabilities are available before committing to a render path
- what optional debug or validation behaviors are actually supported
- what compatibility-relevant limits or format support decisions can be queried explicitly

### Why this pattern fits the release

This direction is compatible with the current architecture because it:

- adds information flow without redefining resource ownership or frame execution
- complements the existing `DXBridge_SupportsFeature()` entry point instead of changing its meaning
- gives multi-language bindings a safer preflight path without forcing a redesign of current examples

### Guardrails for the new family

Any `v1.3.0` capability-query design should follow these implementation constraints:

- prefer new structs over resizing existing ones
- if a query struct is introduced, place `struct_version` at offset `0`
- reserve room for future additive fields rather than planning another near-term breaking shape change
- document whether each query works pre-init, post-init, or per-device
- return information through explicit output buffers or structs, not by overloading existing result codes
- keep failure behavior aligned with the existing `DXBResult` plus thread-local error model

## ABI and API contracts that PR2 must not break

PR2 must treat the following as non-negotiable `1.x` contracts:

1. `include/dxbridge/dxbridge.h` remains the stable public ABI header.
2. `include/dxbridge/dxbridge_version.h` keeps semantic version reporting consistent with the DLL.
3. Existing exports in `src/dxbridge.def` remain stable by both name and ordinal.
4. Existing handles remain opaque 64-bit values.
5. Existing structs keep layout and semantics unchanged.
6. Existing result codes and flag values remain numerically stable.
7. Existing backend-selection and error-retrieval semantics remain unchanged.
8. Existing examples and bindings must continue to work without adopting the new `v1.3.0` feature.

## Design consequences for docs and tests

Compatibility is part of the feature design, so PR2 should include:

- ABI regression coverage for exports, ordinals, and public type assumptions
- tests for additive capability queries without changing old flows
- docs that separate "new optional capability discovery" from "existing stable 1.x rendering flow"
- release notes that clearly call out which items are public additions versus internal hardening

## Deferred to a future major or later minor

The following ideas are intentionally deferred because they would expand scope or increase compatibility risk:

- any redesign that requires changing existing struct layouts
- any attempt to retrofit multiple simultaneous backend instances into the current process model
- any effort to reinterpret existing feature flags or result codes
- large rendering-surface additions unrelated to discovery and compatibility guidance
- changes that would require broad rewrites of all existing language bindings
