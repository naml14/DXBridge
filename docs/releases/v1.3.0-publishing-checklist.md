# DXBridge v1.3.0 Publishing Checklist

Use this checklist when turning the compatible-growth branch into the actual `v1.3.0` release PR.

## 1. Confirm release framing

- `CHANGELOG.md` describes the release in three buckets: public additive value, internal hardening, and documentation or release work
- `docs/releases/v1.3.0-release-notes.md` matches the same message and compatibility guarantees
- `README.md`, `docs/api-reference.md`, `docs/examples.md`, and `docs/integration-guide.md` all present `v1.3.0` as compatible growth, not as a technical-model reset

## 2. Confirm version bump timing

- keep `version.txt`, `CMakeLists.txt`, `include/dxbridge/dxbridge_version.h`, `.release-please-manifest.json`, `README.md`, and `docs/api-reference.md` synchronized at `1.3.0`
- rerun `scripts/verify-version.ps1` after any release-close edit that touches synchronized version metadata
- verify the built DLL also reports `1.3.0` before the final commit/tag handoff

## 3. Run validation commands

From the repository root:

```bat
pwsh -File scripts\verify-version.ps1 -SkipBinaryCheck
cmake --preset debug
cmake --build --preset debug
cmake --build --preset debug --target check_abi
pwsh -File scripts\run-ctest.ps1 -Preset debug -Label smoke
pwsh -File scripts\run-runtime-validation.ps1 -Dll out\build\debug\Debug\dxbridge.dll
```

If a release build is being cut, also run:

```bat
pwsh -File scripts\verify-version.ps1 -Preset release -Configuration Release
```

## 4. Confirm onboarding paths

- `examples/run_capability_preflight.ps1` works for the maintained runtimes you expect to support on the release machine
- Bun, Node.js, Python, Go, and Rust scenario 06 commands still describe the same `--backend`, `--compare-active`, and `--dll` contract
- the integration docs still point new consumers at `DXBridge_QueryCapability()` before `DXBridge_Init()` when preflight is needed

## 5. Confirm release automation assumptions

- Release Please configuration still targets the root package correctly
- `CHANGELOG.md` contains the intended release-ready text before Release Please rewrites the final release notes
- if CI must run on release PRs created by Release Please, use a dedicated token such as `RELEASE_PLEASE_TOKEN` instead of relying only on `${{ github.token }}`
- if AI release-note rewriting is desired, confirm `GEMINI_API_KEY` and optional `GEMINI_MODEL` are configured

## 6. Final publication message

The release PR and GitHub release should both make these points easy to see:

- `v1.3.0` is a compatibility-preserving `1.x` minor
- the only public growth area is additive capability discovery
- existing rendering and backend-lifetime flows stay valid
- the refreshed language examples and runtime validation tooling are part of the release value

## 7. Confirm artifact verification documentation

- `docs/releases/verify-artifacts.md` exists and the asset names it documents match exactly the files uploaded by the `Upload release assets` step in `.github/workflows/releases.yml`
- The documented cosign `--certificate-identity-regexp` pattern matches the workflow path and branch used in the release run
- The documented `gh attestation verify` commands reference the correct repository path
- The README "Verify release artifacts" section links to `docs/releases/verify-artifacts.md` and its asset table matches the current upload list in `releases.yml`
