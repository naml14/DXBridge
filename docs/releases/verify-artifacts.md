# Verifying DXBridge Release Artifacts

Every DXBridge GitHub release is produced by `.github/workflows/releases.yml`.
This document explains what each release asset is, and how to verify it.

## Asset inventory

| Asset | Always present | Purpose |
|---|---|---|
| `dxbridge-v*-win-x64.zip` | Yes | Distribution archive: `dxbridge.dll`, `dxbridge.lib`, and the public headers under `include/dxbridge/` |
| `dxbridge.dll` | Yes | The bare DLL, also uploaded individually so provenance can be attested against it directly |
| `dxbridge.lib` | Yes | Import library for MSVC link-time usage |
| `dxbridge-v*-win-x64.zip.sha256` | Yes | SHA-256 checksum of the ZIP in `sha256sum`-compatible format (`<hex> *<filename>`) |
| `dxbridge-v*-win-x64.manifest.txt` | Yes | Human-readable manifest listing the version, source commit SHA, and files included in the ZIP |
| `dxbridge-v*-win-x64.spdx.json` | Yes | SPDX 2.x Software Bill of Materials (SBOM) for the release build tree, generated by `anchore/sbom-action` |
| `dxbridge-v*-win-x64.zip.cosign.bundle` | No (see note) | Sigstore cosign bundle for keyless verification of the ZIP signature |
| GitHub provenance attestations | No (see note) | GitHub-native build provenance for `dxbridge.dll` and the ZIP, verifiable with `gh attestation verify` |

> **Operational note — transient failures:**
> The cosign signing step and both attestation steps are marked `continue-on-error: true` in
> `releases.yml`. A transient Sigstore or GitHub Attestations service outage will not block the
> release, but it means the bundle and/or attestations may be absent from an otherwise valid
> release. Treat their absence as a service-availability signal, not as evidence of tampering.

## 1. Checksum verification

The `.sha256` file contains a single line in the format expected by `sha256sum`:

```
<lowercase-hex-digest> *dxbridge-v<version>-win-x64.zip
```

### Linux / macOS

```bash
sha256sum -c dxbridge-v<version>-win-x64.zip.sha256
```

Expected output:

```
dxbridge-v<version>-win-x64.zip: OK
```

### Windows (PowerShell)

```powershell
$expected = (Get-Content "dxbridge-v<version>-win-x64.zip.sha256").Split(' ')[0]
$actual   = (Get-FileHash "dxbridge-v<version>-win-x64.zip" -Algorithm SHA256).Hash.ToLowerInvariant()
if ($expected -eq $actual) { "OK" } else { "MISMATCH — do not use this file" }
```

### Windows (certutil)

```bat
certutil -hashfile dxbridge-v<version>-win-x64.zip SHA256
```

Compare the printed digest manually against the value in the `.sha256` file.

## 2. Cosign keyless verification

When the `.zip.cosign.bundle` asset is present, you can verify it with
[cosign](https://github.com/sigstore/cosign).

The release is signed using GitHub Actions OIDC with keyless (Fulcio) signing.
The signing identity is the `releases.yml` workflow in this repository running on the
`refs/heads/master` branch.

```bash
cosign verify-blob \
  --bundle "dxbridge-v<version>-win-x64.zip.cosign.bundle" \
  --certificate-identity-regexp "https://github.com/<owner>/dxbridge/.github/workflows/releases.yml@refs/heads/master" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  "dxbridge-v<version>-win-x64.zip"
```

Replace `<owner>` with the GitHub organization or user that hosts this repository, and
`<version>` with the release version string (e.g. `1.3.1`).

Expected output:

```
Verified OK
```

If the bundle is absent, skip this step. The absence of a cosign bundle does not invalidate
the release — see the operational note above.

## 3. GitHub provenance attestation

GitHub build provenance attestations are generated for both `dxbridge.dll` and the ZIP.
They are stored in the GitHub Attestations service (not as file assets) and are verifiable
with the [GitHub CLI](https://cli.github.com/) (`gh`).

### Verify the ZIP

```bash
gh attestation verify dxbridge-v<version>-win-x64.zip \
  --repo <owner>/dxbridge
```

### Verify the DLL

```bash
gh attestation verify dxbridge.dll \
  --repo <owner>/dxbridge
```

Replace `<owner>/dxbridge` with the full repository path (e.g. `acme/dxbridge`).

Expected output confirms the artifact was built by the `releases.yml` workflow in that
repository. If no attestation is found for a given artifact, the Attestations service may
have had a transient failure at release time — see the operational note above.

## 4. SBOM inspection

The `.spdx.json` file is a standard [SPDX 2.x](https://spdx.dev/) Software Bill of Materials.
It lists the files and packages detected in the release build output by `anchore/sbom-action`.

### Inspect with jq

```bash
# List all package names and versions
jq '.packages[] | {name: .name, version: .versionInfo}' \
  dxbridge-v<version>-win-x64.spdx.json

# Show the document namespace and creation info
jq '{namespace: .documentNamespace, created: .creationInfo}' \
  dxbridge-v<version>-win-x64.spdx.json
```

### Inspect with an SPDX-compatible tool

Any tool that accepts SPDX 2.x JSON format can consume this file.
Examples: [SPDX tools](https://github.com/spdx/tools-java),
[syft](https://github.com/anchore/syft), or a compliance platform that accepts SPDX input.

---

## 5. Manifest inspection

The `.manifest.txt` file is a plain-text summary of the release contents:

```
Version: <version>
Source commit: <sha>
Included files:
- dxbridge.dll
- dxbridge.lib
- include/dxbridge/dxbridge.h
- include/dxbridge/dxbridge_version.h
```

Cross-reference the `Source commit` value against the release tag on GitHub to confirm the
ZIP was built from the expected commit.

## Summary

| What you want to verify | Tool | Asset needed |
|---|---|---|
| File integrity | `sha256sum` / `certutil` / PowerShell | `.zip.sha256` |
| Keyless code signing | `cosign verify-blob` | `.zip.cosign.bundle` (if present) |
| Build provenance | `gh attestation verify` | GitHub Attestations (if present) |
| Software composition | `jq` or SPDX tool | `.spdx.json` |
| Release contents | Text editor | `.manifest.txt` |
