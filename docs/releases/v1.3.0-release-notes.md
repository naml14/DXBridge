# DXBridge v1.3.0 Release Notes

`v1.3.0` is the compatible-growth minor for the DXBridge `1.x` line.

It adds one new public discovery surface, strengthens validation, and updates the release-facing docs without changing the base rendering model, handle model, or process-wide backend model.

## Release headline

DXBridge `v1.3.0` makes backend capability discovery explicit before initialization.

The key public addition is `DXBridge_QueryCapability()`, which lets native and multi-language consumers ask whether DX11 or DX12, debug layers, GPU validation, and adapter-specific capabilities are available before they commit to `DXBridge_Init()`.

## Public additive value

- add `DXBridge_QueryCapability()` as an additive export appended after the existing v1 baseline
- add `DXBCapabilityQuery` and `DXBCapabilityInfo` so callers can retrieve capability answers without changing existing structs
- add capability IDs for backend availability, debug layer availability, GPU validation availability, adapter count, software-adapter detection, and adapter max feature level
- add onboarding-ready helper coverage and scenario 06 examples in Bun, Node.js, Python, Go, and Rust

## Internal hardening

- expand ABI-oriented validation so export ordinals, type widths, and public struct assumptions stay guarded
- expand capability-focused regression coverage around pre-init queries and adapter-scoped validation
- add cross-runtime capability validation tooling so the maintained bindings can be smoke-checked against one built DLL

## Documentation and release work

- refresh `README.md` so the compatible `v1.3.0` value is easy to understand from the repository entry point
- refresh `docs/api-reference.md`, `docs/examples.md`, and `docs/integration-guide.md` so capability discovery is documented as an additive path rather than a contract reset
- refresh the affected language READMEs so scenario 06 is positioned as the new onboarding path while scenarios 01 through 05 remain the stable `1.x` flow
- keep `CHANGELOG.md` and the publishing checklist aligned for the final release handoff

## Compatibility statement

`v1.3.0` is a minor release for the existing `1.x` ABI.

It does not:

- remove or rename existing `DXBridge_*` exports
- change existing export ordinals
- resize or reorder existing public structs
- change result-code, flag, enum-like, or handle-width values
- redefine `DXBridge_Init()` or `DXBridge_SupportsFeature()` semantics

Existing consumers can keep their current flow unchanged. The capability-query family is optional.

## Recommended consumer message

If you already use DXBridge `1.x`, you only need `v1.3.0` when you want better pre-init backend discovery or clearer onboarding across the maintained language bindings. If you do not need that, your existing integration model remains valid.
